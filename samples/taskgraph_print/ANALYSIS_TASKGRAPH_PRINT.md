# Overview
The `taskgraph_print` directory provides a minimal MPI-based utility for introspecting Astaroth's default task graph structure. It initializes the GPU subsystem, retrieves the default task graph via `acGridGetDefaultTaskGraph()`, and writes the graph's dependency information to a per-process text file (`dependencies_pid_<rank>.txt`). A commented-out alternative shows the capability to also write the full task list via `acGraphWriteTasks()`. The program is intended for development and debugging — by examining the generated dependency files, developers can verify that the task graph construction, dependency resolution, and subregion partitioning are working correctly.

# Program Flow

## 1. MPI Initialization
- `MPI_Init(NULL, NULL)`
- `MPI_Comm_rank(MPI_COMM_WORLD, &pid)` — get process rank.

## 2. Configuration & GPU Initialization
- `acLoadConfig(AC_DEFAULT_CONFIG, &info)` — load mesh configuration.
- `acGridInit(info)` — initialize GPU subsystem, create global grid and **default task graph**.

## 3. Default Task Graph Retrieval
- `acGridGetDefaultTaskGraph()` — retrieve the default task graph that was created during `acGridInit()`. This is the task graph generated by the DSL system (`acGetDSLTaskGraph`) when the grid is initialized, representing the default physics solver pipeline.

## 4. Dependency File Writing
- Construct filename: `"dependencies_pid_" + std::to_string(pid)` — per-process output file named by MPI rank.
- `acGraphWriteDependencies(dependencies_file_path.c_str(), default_tasks)` — write the task graph's dependency information to the file.
- **Commented-out alternative:** `acGraphWriteTasks(tasks_file_path.c_str(), default_tasks)` would write the full task list (including task names, fields, kernels, etc.) instead of just dependencies.

## 5. Cleanup
- `acGridQuit()` — shutdown GPU subsystem.
- `MPI_Finalize()` — shutdown MPI.
- `return EXIT_SUCCESS` — clean exit.

# Task Graph Introspection

## `acGraphWriteDependencies`
This function outputs the dependency structure of the task graph — which tasks depend on which, in what order they execute, and how subregions relate to each other. The output is useful for:
- Verifying that halo exchange tasks precede compute tasks.
- Checking that boundary condition tasks are correctly ordered.
- Understanding how tasks are partitioned across subregions.
- Debugging deadlock or ordering issues in custom task graphs.

## `acGraphWriteTasks` (Commented Alternative)
This function would output the full task list including:
- Task names and types (halo exchange, boundary condition, compute).
- Associated kernels.
- Input/output field arrays.
- Parameters passed to kernels.
- Spatial bounds for each task.

# Notable Observations

1. **Smallest sample in the suite:** At 40 lines, this is the most concise example — only 5 non-trivial lines of actual logic (MPI init, config load, grid init, task graph get, dependency write, quit, finalize).

2. **Pure introspection tool:** The program does not execute any physics or perform any computation. It solely inspects and dumps the default task graph structure, making it a development/debugging utility rather than a physics solver or benchmark.

3. **Per-process output files:** Each MPI rank writes to its own file (`dependencies_pid_<rank>.txt`), reflecting the fact that each process has its own partition of the domain and therefore its own local view of the task graph's subregion decomposition.

4. **Default task graph source:** The "default task graph" is the one created internally by `acGridInit()` — it is generated from the DSL system (`acGetDSLTaskGraph`) based on the DSL file loaded during initialization. This is the same task graph that would be executed by `acGridExecuteTaskGraph()` without arguments.

5. **Dependency vs. task list distinction:** The program writes only dependencies (not the full task list), suggesting the primary debugging need is understanding task ordering rather than inspecting full task contents. The commented-out `acGraphWriteTasks()` call implies both functions exist and serve different introspection purposes.

6. **No file format documentation:** The exact output format of `acGraphWriteDependencies()` is not documented in the example. It likely outputs a human-readable text representation of the dependency graph (possibly DOT/graphviz format, or a simple list of (task, dependencies) pairs).

7. **No error checking:** The program does not check the return value of `acGridGetDefaultTaskGraph()` (which could return NULL if no default task graph exists) or `acGraphWriteDependencies()` (which could fail if the file cannot be written).

8. **No host mesh creation:** Unlike most other samples, this program never creates a host mesh (`acHostMeshCreate`). It only needs the mesh configuration (`AcMeshInfo`) to initialize the grid and construct the task graph.

9. **No mesh loading:** The program does not call `acGridLoadMesh()` — it initializes the grid infrastructure and introspects the task graph without actually transferring any mesh data to the GPU.

10. **No execution:** The task graph is retrieved but never executed (`acGridExecuteTaskGraph` is not called). This is appropriate for an introspection tool — it is dumping the graph structure, not running it.

11. **File path construction uses std::string:** The per-process filename is built with `std::string` concatenation and `std::to_string()`, requiring `#include <string>`. This is a simple but effective approach for unique per-process output filenames.

12. **`astaroth_debug.h` included but not obviously used:** The `#include "astaroth_debug.h"` header on line 2 is present but no debug-specific API is called in the visible code. It may be that `acGraphWriteDependencies` and `acGraphWriteTasks` are declared in this header, or it was included by habit from other samples.

13. **`errchk.h` included but unused:** Similarly, `#include "errchk.h"` is included but no error checking macros are used in the program. This suggests the header may be a common inclusion pattern across samples.

14. **Exit code always EXIT_SUCCESS:** The program always returns `EXIT_SUCCESS` (line 29), even if `acGraphWriteDependencies` might have failed. This is consistent with the minimal error handling throughout the program.

15. **`#endif` typo:** Line 40 has `#endif // AC_MPI_ENABLES` (missing the final `D` in `ENABLED`), which is a cosmetic comment typo. The preprocessor directive on line 6 correctly uses `AC_MPI_ENABLED`.

16. **Config-driven DSL graph:** The default task graph content depends entirely on what DSL file `AC_DEFAULT_CONFIG` points to. Different configurations will produce different task graphs (different kernels, different boundary conditions, different field arrays).

17. **No output to stdout:** Unlike `taskgraph_example` or `standalone`, this program does not print any progress messages to `std::cout` or `printf`. All output goes to files. The lack of stdout output makes it silent unless the user checks the generated dependency files.

18. **`acGridGetDefaultTaskGraph` returns a pointer:** The function returns `AcTaskGraph*`, suggesting the default task graph is owned by the grid infrastructure and should not be manually destroyed — calling `acGridDestroyTaskGraph` on it could corrupt the grid state.

19. **Single-stream execution:** The program operates on the default stream (no explicit stream specification), meaning all introspection operations (file writes) are synchronous with respect to GPU execution.

20. **Utility function candidates:** `acGraphWriteDependencies` and `acGraphWriteTasks` appear to be first-class debugging utilities in the Astaroth library, likely intended for ongoing development use. They could also be useful for automated testing — comparing expected vs. actual dependency structures.
