# Overview
The `pc-varfile-import` sample is an MPI-accelerated mesh initialization utility for Astaroth that reads initial condition data from an external varfile format (produced by a separate simulation code) and loads it into Astaroth's GPU grid infrastructure. It reads all specified fields from a single binary file (`var.dat`), populates the GPU mesh, computes per-field statistics (min/RMS/max), writes Astaroth-compatible snapshots to disk, and exports per-slice data for visualization. The sample is designed for initializing large cosmological/MHD simulations (default grid: 4096Âł) with data generated by an external code (e.g., a 3D N-body or MHD solver).

# Program Flow

## 1. MPI Init
`MPI_Init`, get rank (`pid`) and total processes (`nprocs`).

## 2. Field Array Definition
```c
const Field fields[] = {
    VTXBUF_UUX,
    VTXBUF_UUY,
    VTXBUF_UUZ,
    VTXBUF_LNRHO,
#if LMAGNETIC
    VTXBUF_AX, VTXBUF_AY, VTXBUF_AZ,
#endif
};
```
4 fields for hydro (velocity + density), 7 if `LMAGNETIC` is defined (adds magnetic vector potential).

## 3. Mesh Configuration
a. `acLoadConfig(AC_DEFAULT_CONFIG, &info)` â€” load default mesh config.
b. `info.int3_params[AC_ngrid] = nn` â€” override global grid dimensions to 4096Âł.
c. `acHostUpdateParams(&info)` â€” process updated parameters.

## 4. Grid Init & Varfile Read
a. `acGridInit(info)` â€” initialize GPU grid infrastructure.
b. `acGridReadVarfileToMesh(file, fields, num_fields, nn, rr)` â€” read the varfile and populate the GPU mesh. This is the core operation of the sample.

## 5. Per-Field Statistics
For each of `NUM_VTXBUF_HANDLES` vertex buffers (all fields, not just the imported ones):
a. `acGridReduceScal(STREAM_DEFAULT, RTYPE_MAX, i, &buf_max)` â€” compute max.
b. `acGridReduceScal(STREAM_DEFAULT, RTYPE_MIN, i, &buf_min)` â€” compute min.
c. `acGridReduceScal(STREAM_DEFAULT, RTYPE_RMS, i, &buf_rms)` â€” compute RMS.
d. `printf("  %*s: min %.3e, rms %.3e, max %.3e\n", ...)` â€” print statistics with field name.

## 6. Output Directory Creation
a. `job_id = 12345` (hardcoded).
b. `snprintf(job_dir, 4096, "output-%d", job_id)` â†’ `"output-12345"`.
c. `system("mkdir -p output-12345")` â€” create output directory.

## 7. Astaroth Mesh Snapshot Write
For each of `NUM_VTXBUF_HANDLES` fields:
`acGridAccessMeshOnDiskSynchronous((VertexBufferHandle)i, job_dir, vtxbuf_names[i], ACCESS_WRITE)` â€” write each field to `job_dir/field_name` using Astaroth's MPI-IO subsystem.

## 8. Slice Export
a. `acGridWriteSlicesToDiskLaunch(job_dir, 0, 0.0)` â€” launch async write of X-Y slices (constant-Z planes) to `job_dir/slices-<step>.dat`.
b. `acGridDiskAccessSync()` â€” wait for async slice write to complete.

## 9. Merge Slices
Call `merge_slices(job_dir, 0, nn.x, nn.y, fields, num_fields)`:
a. Allocate buffer of size `nx * ny` (one X-Y plane).
b. For each imported field:
   1. Open `job_dir/<field_name>-000000.slice` (single slice file).
   2. Read plane data into buffer.
   3. Write plane data to `job_dir/slices`.
c. Close files and free buffer.

## 10. Cleanup
`acGridQuit()`, `MPI_Finalize()`. If `pid == 0`, print `"OK!"`.

# Varfile Format

The varfile (`var.dat`) is a binary file containing interleaved field data in row-major (C-style) order. Each field is a contiguous array of `nn.x * nn.y * nn.z` `AcReal` values. Fields are stored sequentially:

```
[field0_data] [field1_data] [field2_data] ... [fieldN_data]
```

Where each field is:
```
data[i + j*nn.x + k*nn.x*nn.y]  // for i=0..nn.x-1, j=0..nn.y-1, k=0..nn.z-1
```

The `rr` parameter (stencil radius) defines the ghost cell extent â€” the imported data occupies the "interior" of a larger mesh with `rr` ghost cells on each side.

# Slice Format

Slices are exported as 2D X-Y planes at constant Z. Each slice file is named:
```
job_dir/<field_name>-<label>.slice
```
Example: `output-12345/VTXBUF_UUX-000000.slice`

The `merge_slices` function reads each field's slice file and concatenates all planes into a single `job_dir/slices` file.

# Notable Observations

1. **Hardcoded HPC path**: The varfile path `/scratch/project_462000077/mkorpi/forced/mahti_4096/data/allprocs/var.dat` is specific to the Mahti HPC cluster (project 462000077 at CSC, Finland). This makes the sample unusable without modification on other systems. A command-line argument for the input path would be a significant improvement.

2. **Stencil radius `rr = (3, 3, 3)`**: The stencil radius of 3 means each kernel accesses a 7Âł neighborhood. This is unusually large â€” most MHD kernels use `rr = 1` (3Âł neighborhood) or `rr = 2` (5Âł neighborhood). A radius of 3 suggests either a high-order stencil or a multi-step scheme.

3. **4096Âł grid**: The default grid size is 4096Âł = 68.7 billion cells. At 8 bytes per real (double precision), a single field requires ~522 GB of GPU memory. For 4 hydro fields, that's ~2 TB. This sample assumes a multi-GPU setup with massive memory. The grid must be decomposed across many MPI ranks.

4. **Statistics computed for ALL fields, not just imported ones**: The per-field statistics loop iterates over `NUM_VTXBUF_HANDLES` (all DSL fields), not just the imported fields. Uninitialized fields (those not in the varfile) will show min/max/RMS of zero or NaN, which could be confusing.

5. **`LMAGNETIC` warning, not error**: If `LMAGNETIC` is not defined, only a `WARNING()` is printed â€” the program continues with 4 fields instead of 7. This allows the same binary to be used for both hydro and MHD simulations, but the warning is easy to miss.

6. **`merge_slices` writes to a single file**: The `merge_slices` function opens `job_dir/slices` (no field name in the output path) and writes all field data sequentially. This creates a monolithic file containing all imported fields' slice data concatenated together. The file has no headers or field separators â€” a consumer would need to know the field count and order to parse it.

7. **Slice label is always 0**: `acGridWriteSlicesToDiskLaunch(job_dir, 0, 0.0)` always uses step 0 and time 0.0. This is appropriate for an initialization import where no time evolution has occurred yet.

8. **No error checking on reductions**: The `acGridReduceScal` calls that compute per-field statistics have no error checking. A failed reduction would produce garbage statistics.

9. **Output directory via `system()`**: The `mkdir -p` command is executed via `system()`, which runs on all MPI ranks. With 4096+ ranks, this could cause race conditions if multiple ranks simultaneously try to create the same directory. In practice, `mkdir -p` is idempotent, so this is safe, but it's still redundant.

10. **Synchronous I/O for snapshots, async for slices**: The mesh snapshot write uses synchronous `acGridAccessMeshOnDiskSynchronous`, while the slice export uses the async `acGridWriteSlicesToDiskLaunch` + `acGridDiskAccessSync` pattern. This inconsistency suggests two different I/O APIs were available and used interchangeably.

11. **No data verification**: Unlike `mpitest`, there is no CPU-side reference to compare against. The sample trusts that `acGridReadVarfileToMesh` correctly imports the varfile data. A useful extension would be to verify the import by reading back a field and comparing against the original varfile.

12. **`acGridReadVarfileToMesh` is DSL-loaded**: The function is loaded via `LOAD_DSYM` at runtime from the DSL-compiled library (see `astaroth_grid.h` line 165 and `astaroth.h` line 551). This means the varfile reader is part of the DSL-generated code, allowing it to understand DSL-specific field layouts.

13. **No MPI error checking**: Like other MPI samples, MPI calls have no error checking. `MPI_Init`, `MPI_Comm_rank`, `MPI_Comm_size`, and `MPI_Finalize` could fail silently.

14. **Commented-out debug code**: Lines 102-151 contain a large commented-out section that generates a synthetic test varfile using `powf(i*i + j*j, 0.5)` and writes it with `fwrite`. This could be useful for testing without an external varfile.

15. **Field order in varfile**: The varfile format stores fields in the order specified by the `fields[]` array: UUX, UUY, UUZ, LNRHO (and AX, AY, AZ if `LMAGNETIC`). The varfile reader assumes this exact order.

16. **C compilation, not C++**: The sample uses `.c` (C) rather than `.cc` (C++). This is one of the few MPI samples that doesn't use C++ features (no `std::vector`, no `std::printf` â€” pure C throughout).

17. **Math library linked**: `-lm` is linked via `target_link_libraries(... m)`. This is needed for the commented-out debug code (`powf`), but not for the active code paths.

18. **No timing or benchmarking**: The sample measures nothing â€” no timers, no bandwidth calculations. It's purely a utility for initializing simulation data.

19. **Job ID hardcoded**: `job_id = 12345` means the output directory is always `output-12345`. Re-running the sample would overwrite previous output. A command-line argument or timestamp-based directory name would be safer.

20. **`merge_slices` buffer allocation per field**: The `merge_slices` function allocates a single `count = nx * ny` buffer and reuses it for all fields. This is memory-efficient but means only one field's data is in memory at a time during the merge.
