# Overview
The `heat-equation` sample is a comprehensive benchmarking suite for stencil-based heat equation solvers on GPU hardware. It consists of:

1. **`main.c`**: A standalone Astaroth-native heat-equation solver benchmark using Astaroth's kernel launch infrastructure, supporting stencils of orders 0, 2, 4, 6, and 8 in 1D, 2D, and 3D.
2. **`heat-equation.ac`**: An Astaroth compiler definition file (generated by `gen-heat-equation-kernel.py`) that defines the `temperature` field and 45 stencil/kernel combinations.
3. **`heat-equation.py`** and **`heat-equation-separable.py`**: Python benchmarking scripts for PyTorch, TensorFlow, and JAX backends, with correctness verification against a SciPy reference.
4. **`cudnn-heat-equation/`**: A cuDNN standalone benchmark (2D convolution, 4096×4096 input) using native CUDA/cuDNN APIs.
5. **`miopen-heat-equation/`**: An AMD MIOpen equivalent of the cuDNN benchmark with `#define` shims mapping cuDNN API calls to MIOpen equivalents.

The suite is designed to compare Astaroth's performance against state-of-the-art deep learning frameworks (PyTorch, TensorFlow, JAX, cuDNN, MIOpen) on the same mathematical problem — second-order spatial derivative (Laplacian) discretization via finite differences.

# Program Flow (main.c)

1. **Parse arguments** from `argv` with defaults.
2. **Print implementation info**: `IMPLEMENTATION`, `MAX_THREADS_PER_BLOCK`, `STENCIL_ORDER`, `DOUBLE_PRECISION`.
3. **Determine kernel name**: `solve{N}d_ord{2R}` where N = 1/2/3 dimensions, R = radius.
4. **Mesh configuration**: `acLoadConfig(AC_DEFAULT_CONFIG, &info)` → `acSetMeshDims(nx, ny, nz, &info)`.
5. **Device creation**: `acDeviceCreate(0, info, &device)`, `acDevicePrintInfo(device)`.
6. **Random number setup**: `acRandInitAlt(seed, count, pid)` + `srand(seed)`.
7. **Stencil coefficients**: `load_stencil_coefficients(device, dims, radius)` — constructs 1D/2D/3D stencils with finite-difference weights, normalizes with `dt`, spatial step sizes, and identity center term, then uploads to device via `acDeviceLoadStencils`.
8. **Verification mode** (if `verify == 1`):
   - Run a dryrun kernel launch and `acDeviceResetMesh`.
   - **Load/store check**: randomize → load to device → store back → `acVerifyMesh("Load/Store", ...)`.
   - **Boundary condition check**: apply constant bounds (0.0) → load → store → `acVerifyMesh("Boundconds", ...)`.
   - **Integration check**: run `num_verification_steps` (3) integration steps on both device and host (`acHostSolveHeat`), then `acVerifyMesh("Kernel", ...)`.
9. **Benchmark loop** (`num_samples` iterations):
   - Dryrun: launch kernel → reset mesh → randomize → swap buffers → synchronize.
   - Measure: reset timer → launch kernel → synchronize → record milliseconds.
   - `acDeviceBenchmarkKernel(device, kernel, dims.n0, dims.n1)` — detailed profiling.
   - Write CSV row: kernel name, implementation ID, max tpb, nx/ny/nz, radius, milliseconds, tpb.x/y/z, jobid, seed, iteration, double_precision.
10. **Print final result** for last iteration: milliseconds and optimal thread block dimensions.
11. **Cleanup**: close CSV file, destroy device, free host meshes.

# Stencil Coefficients (Finite-Difference Weights)

The sample uses compact central-difference stencils for second-order derivatives:

| Radius | Order | Coefficients | Formula |
| :--- | :--- | :--- | :--- |
| 0 | 0 | `{0.}` | Identity (no-op) |
| 1 | 2 | `{1, -2, 1}` | Standard 3-point Laplacian |
| 2 | 4 | `{-1/12, 4/3, -5/2, 4/3, -1/12}` | 5-point compact stencil |
| 3 | 6 | `{1/90, -3/20, 3/2, -49/18, 3/2, -3/20, 1/90}` | 7-point compact stencil |
| 4 | 8 | `{-1/560, 8/315, -1/5, 8/5, -205/72, 8/5, -1/5, 8/315, -1/560}` | 9-point compact stencil |

# Astaroth Compiler Definition (`heat-equation.ac`)

Defines 45 kernel/stencil combinations:

| Dimension | Radii | Stencils | Kernels |
| :--- | :--- | :--- | :--- |
| 1D | 0, 1, 2, 3, 4 | `heat1d_ord0` … `heat1d_ord8` (5) | `solve1d_ord0()` … `solve1d_ord8()` (5) |
| 2D | 0, 1, 2, 3, 4 | `heat2d_ord0` … `heat2d_ord8` (5) | `solve2d_ord0()` … `solve2d_ord8()` (5) |
| 3D | 0, 1, 2, 3, 4 | `heat3d_ord0` … `heat3d_ord8` (5) | `solve3d_ord0()` … `solve3d_ord8()` (5) |
| **Total** | | **15 stencils** | **15 kernels** |

Each stencil applies the FD coefficients along its dimension(s) with identity on the center point.

# Host-Side Heat Solver (`acHostSolveHeat`)

A reference CPU implementation of the same finite-difference stencil used for verification:

1. Creates an output mesh.
2. Iterates over the domain `[n0, n1)` in Z, Y, X order.
3. For each cell, applies the stencil sum along each active dimension (X, Y, Z), weighted by `dt / ds²` where `ds = 2π / nn`.
4. Copies results back to the input mesh.

# Low-Level CUDA/cuDNN/MIOpen Benchmarks

## cuDNN Benchmark (`cudnn-heat-equation/main.cu`)

- **Problem**: 2D convolution, 4096×4096 input, 3×3 kernel, NCHW format, float32.
- **Algorithm selection**: `cudnnFindConvolutionForwardAlgorithm` — auto-tunes to find fastest algorithm.
- **Warmup**: 10 iterations before timing.
- **Output**: prints "Time elapsed: X ms" via `timer_hires.h`.
- **Verification**: fills input/filter with 1.0, verifies output == 9 (3×3 kernel of all-ones convolved with all-ones).

## MIOpen Benchmark (`miopen-heat-equation/main.cu`)

- Same structure as cuDNN benchmark but with `#define` shim layer mapping cuDNN → MIOpen API calls.
- Uses `hip.h` for CUDA → HIP type/function aliases.
- Algorithm selection: `miopenFindConvolutionForwardAlgorithm` with exhaustive search.
- Outputs selected algorithm: GEMM, Direct, FFT, Winograd, or ImplicitGEMM.

# Notable Observations

1. **Cross-framework benchmarking**: The entire suite is designed to produce comparable performance numbers across Astaroth (native), cuDNN (NVIDIA), MIOpen (AMD), PyTorch, TensorFlow, and JAX — all solving the same heat equation stencil.

2. **Two-tier verification**: In `verify` mode, the sample performs three levels of correctness checks: (a) device round-trip, (b) boundary condition consistency, (c) full kernel integration against a sequential CPU finite-difference solver. This is a rigorous approach to GPU kernel validation.

3. **Astaroth DSL compiler module**: The `build.sh` sets `DSL_MODULE_DIR` and `PROGRAM_MODULE_DIR` to point at this sample's directory, suggesting `heat-equation.ac` is loaded as a custom Astaroth compiler module during the build — not a standard library module.

4. **cuDNN → MIOpen portability**: The `miopen-heat-equation/` benchmark uses a `#define` shim layer to compile the exact same C code against either cuDNN or MIOpen. The `hip.h` header provides an additional CUDA → HIP type/function alias layer.

5. **Large benchmark grid**: The standalone cuDNN/MIOpen benchmarks use 4096×4096 inputs — 4× larger than the default 256³ Astaroth benchmark — suggesting these are focused on measuring raw convolution library throughput rather than application-scale performance.

6. **Stencil order scaling**: Support for orders 0 through 8 (radii 0–4) means stencil widths of 1, 3, 5, 7, and 9 points respectively — enabling systematic accuracy vs. performance analysis.

7. **Timestep formula**: The Python scripts use `dt = 1e-3 * min(dx, min(dy, dz))` for stability (CFL-like condition), while `main.c` uses `FLT_EPSILON` (essentially zero) to benchmark kernel overhead without significant solution change.

8. **README comparison**: The README includes a side-by-side comparison showing Astaroth at ~20k M elements/sec vs. PyTorch `conv3d` at ~3.8 M elements/sec — a ~5000× difference that reflects both the simplicity of the kernel and the overhead of PyTorch's initial (unoptimized) implementation noted in the comment "no optimizations".

9. **Seeded determinism**: Random seeds combine `salt + time + dimension parameters` for reproducible-but-different runs across CI executions.
